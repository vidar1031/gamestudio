# 产品工具分析（2026-03-24）

面向对象：`game_studio`（点击交互小故事制作工具，P0 已闭环，P1 目标是稳定生产）。

---

## 1. 结论（TL;DR）

- 当前工具链已达到 **P0 可用闭环**：三层工作流（脚本→蓝图→合成）可跑通，项目可预览与导出 H5；AI 文本与图像链路已接入多 Provider，且已有最小 smoke test 入口（`smoke_test_pipeline.sh`）。
- 但距离“稳定生产”仍有明确短板，主要集中在三类瓶颈：
  1) **多 Provider 的可用性不可预测**（连通性/深度可用/模型差异/超时策略），导致成功路径对用户不够确定；
  2) **批量与长任务的治理不足**（队列化、暂停/取消/续跑、失败分层与可观测性仍需统一）；
  3) **连续性与资产引用链路仍脆弱**（尤其“参考图必须是公网可访问 URL / remoteUrl 才能作为云端参考”的客观约束，影响历史资产复用与稳定复现）。

结论：工具本身方向正确、架构边界清晰（editor/server/builder/runtime/schema），下一阶段不应追求“功能扩张”，而应以“可预测的成功路径 + 可恢复的失败路径 + 可验证的验收基线”为核心交付。

---

## 2. 产品定位与边界

### 2.1 产品定位

- 目标：把“故事主题”稳定转换为 **可试玩、可导出、可复用** 的 H5 交互小故事产物。
- 核心生产链路（北极星）：故事创建 → 脚本转蓝图 → 连续性约束 → 场景图生成 → 合成与导出 → 测试与验收。

### 2.2 明确边界（保持收敛）

- P1 阶段应优先做：工程治理、稳定性、可扩展接口（而非 UI 大改/玩法扩张/多引擎切换）。
- 主价值不是“做一个能编辑的工具”，而是“做一个可稳定交付产物的产线工具”。

---

## 3. 关键用户与 JTBD

### 3.1 典型角色

- 内容策划/编剧：写脚本、调结构、看分支收束是否合理。
- 技术策划：检查蓝图结构、不可达、占位与合成约束。
- 美术/制作：生成或替换场景图、保证角色/画风一致。
- 运营/交付：导出 H5、验收可玩性、复用样例模板。
- 工程/AI 调参：Provider 配置、诊断、超时/重试/日志定位。

### 3.2 JTBD（最小可用成功定义）

在 30~60 分钟内完成：
1) 生成或编辑 `scripts.json`（有明确选择点/后果/结局）；  
2) 编译并验证 `blueprint.json`（无关键结构错误）；  
3) 合成生成可运行 `story.json`，绑定必要资源；  
4)（可选）生成关键节点背景图且连续性稳定；  
5) 导出 H5，打开即可试玩并覆盖关键分支。

## 4. 工具链现状（系统构成）

### 4.1 组件与职责边界

- Editor（`apps/editor`，Vite + React + TS）：工作台 + 三层工作流 UI、设置面板、预览/导出入口。
- Server（`apps/server`，Hono，1999）：文件项目制存储、编译/导出、AI 调用与诊断、资源写入。
- Schema（`packages/schema`）：数据结构与迁移/约束承载（稳定性与兼容关键）。
- Builder（`packages/builder`）：插件化导出管线与产物协议（P0：`story-pixi`）。
- Runtime（`packages/runtime-pixi`）：导出产物的可运行渲染执行。
- Storage（`storage/` 或 `STUDIO_STORAGE_ROOT`）：`projects/<projectId>` 目录即“项目数据库”。

### 4.2 数据与工作流载体（文件项目制）

项目目录关键文件（建议视为“产品协议”）：

- `scripts.json`：脚本层草稿（分镜卡片）。
- `blueprint.json`：蓝图层结构（节点/分支/占位）。
- `project.json` / `meta.json`：项目元信息与生成公式/配置快照。
- `story.json`：合成层可运行故事（运行时的主要输入）。
- `assets/`：素材（图/音频等，含本地路径与可能的 `remoteUrl`）。

### 4.3 AI 配置与可用性机制

- 配置优先级：`storage/_config/studio_settings.json` > `.env.local` > `.env` > 默认值。
- Provider 维度分离：`scripts` / `prompt` / `image` / `tts`（可分别选择 provider/model/url/key）。
- 关键诊断入口：`POST /api/ai/diagnose`（用于“能不能跑”的基线判断）。

## 5. 端到端主流程（建议标准“成功路径”）

> 目的：让用户始终知道“下一步是什么”，并让失败具备明确可恢复路径。

1) 设置页完成 provider/model/key，并跑一次诊断（确保文本/图像链路可用）。  
2) Hub 新建项目或 AI 创建项目（得到 `scripts.json` 草稿）。  
3) Script Studio：按约定完善选择点/后果/结局（优先修结构与文案）。  
4) Blueprint Studio：编译 + 校验（修不可达、缺后果、无可达结局等）。  
5) Compose Studio：结构不变前提下绑定资源、预览运行。  
6)（可选）批量分镜/单场景出图：以“连续性 Bible + 场景增量”方式推进；失败可续跑。  
7) 导出 H5：打开 `server /demos/.../dist/index.html` 试玩；形成可交付物。

建议把 `smoke_test_pipeline.sh` 的验证路径视为“产线最低验收标准”，并持续保持它可以在任意时间复现“成功路径”。

## 6. 现状评估（按生产链路）

| Stage | 当前结论 | 主要风险 | 优先级建议 |
|---|---|---|---|
| 故事创建（scripts） | 可用（支持 AI/手写，结构公式可保存） | AI 成功率与时延波动、脚本格式偏差导致后续返工 | A |
| 脚本→蓝图 | 可用（编译+校验+解释） | 结构约定不一致导致“不可达/缺后果”频发；错误提示需更面向用户 | A |

## 7. 关键问题（产品视角的根因诊断）

### 7.1 “成功路径”对用户仍不够可预测

- 现象：同一套操作在不同环境/不同 provider 下体验差异大（超时、参数不兼容、模型限制）。
- 根因：Provider 的“连通性 ok”与“深度可用 ok”未形成统一标准；超时/重试/错误码虽在推进，但仍需要一条可解释的产品化门禁。

### 7.2 长任务与批量任务仍缺“生产级治理”

- 现象：批量提示词/批量出图虽然可续跑，但缺少系统级队列/并发/暂停/取消的统一语义与观测面板（尤其跨会话/跨重启）。
- 根因：产线任务从“功能实现”迈向“调度系统”需要明确的任务模型与状态机（P1/M2 的核心）。

### 7.3 连续性与参考资产存在客观约束，需要产品化呈现

- 现象：云端图片生成要用参考图时，必须是可被云端访问的 URL；本地 `localhost` 或仅本地文件无法作为参考。
- 根因：外部服务的输入约束；工具需要把“哪些资产可作为参考、为什么不可、怎么修复”变成稳定的产品规则与操作指引。

## 8. 专业结论：优先级建议（带验收标准）

### A0（最高优先）：固化“可预测成功路径”

- 交付目标：任何新环境下，用户能在 10 分钟内完成“配置→诊断→跑通样例→导出试玩”的闭环。
- 验收标准：
  - 一键诊断能明确给出：缺哪个 key/哪个 URL 不通/哪个模型参数不兼容；
  - 给出“下一步怎么修”的可执行指引（而不是仅报错）。

### A1：Provider 健康检查标准化（连通性 vs 深度可用）

- 交付目标：把 provider 不稳定从“玄学”变成“可测、可解释、可回归”。
- 验收标准：
  - 同一份诊断输出在 UI 与 API 表达一致（provider/model/latency/错误分类/建议降级）。

### A2：任务队列化与可恢复（提示词/出图）

- 交付目标：批量任务具备可暂停、可取消、可续跑的统一语义，重开编辑器仍可恢复推进。
- 验收标准：
  - 批量任务在任意阶段失败时，可在不丢已有结果的前提下恢复完成；
  - 任务状态可追踪（traceId、阶段耗时、失败原因、重试次数）。

### A3：金样例项目与回归基线

- 交付目标：以“产物可玩”为中心的回归，不再依赖人工记忆。
- 验收标准：
  - 至少 1~3 个 `demo_library` 金样例可稳定跑通 smoke test；
  - 关键输出（结构、导出可访问性）可自动验收。

### A4：参考资产的 remoteUrl 策略产品化

- 交付目标：把“参考图能不能用”变成明确规则与工具能力（而不是隐性约束）。
- 验收标准：
  - UI 显式标注“可用作云端参考/不可用”的原因；
  - 提供最小可行的修复路径（例如重新生成一次以落 remoteUrl，或提供上传/中转策略）。

---

## 9. 建议指标（用于形成数据闭环）

- TTFP（Time to First Playable）：从新建项目到可试玩导出的耗时。
- Stage 成功率：scripts 生成、蓝图编译、合成、出图、导出各阶段成功率。
- 可恢复效率：失败后到恢复完成的平均耗时（MTTR）。
- 资源稳定性：角色/画风漂移在连续场景中的“复现率/漂移率”（可先以人工标注+日志近似）。
- 诊断覆盖率：诊断能否命中并解释真实失败原因（命中率）。

---

## 10. 下一步（建议落地顺序）

1) 把 `smoke_test_pipeline.sh` 作为每日/每次推进的必跑基线，并沉淀“失败类型→修复动作”清单。  
2) 完成 Provider 健康检查与诊断输出的产品化（同一份信息同时服务 UI 与 API）。  
3) 以“批量分镜”为中心推进队列化/可恢复能力，作为稳定生产的抓手。
