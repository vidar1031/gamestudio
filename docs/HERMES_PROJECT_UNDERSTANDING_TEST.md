# Hermes 项目理解测试

本文档用于验证 Hermes 是否已经足够理解 GameStudio 当前项目结构、功能边界和执行路径，并判断它适合做哪一类任务。

## 结论先行

不要靠不断闲聊来赌它“慢慢学会项目”。

正确顺序是三段式：

1. 问答校准
2. 小任务执行
3. 长任务 plan

只有前一段稳定通过，才进入后一段。

## 为什么不能直接上长 plan

如果 Hermes 还不能稳定回答以下事实，长 plan 只会建立在错误前提上：

- 目录在哪
- 前后端分别在哪
- 项目创建目录在哪
- control / hermes / server 的责任边界是什么
- stories 扫描应该优先扫哪里

所以当前阶段不应该先做“很长的计划任务”，而应该先做“项目事实理解测试”。

## 三段式测试法

### 第一段：问答校准

目的：验证 Hermes 是否读对了权威文档和项目记忆。

题型要求：

- 必须是有标准答案的问题
- 答案必须能落到文件和目录事实
- 不允许开放式发挥

通过标准：

- 10 题中至少 8 题正确
- 回答中不能把不存在的目录或功能说成存在
- 关键路径不能答错：`apps/editor`、`apps/server`、`apps/control-console`、`apps/control-server`、`storage/projects`

推荐题目：

1. GameStudio 的编辑器前端目录在哪里？
2. GameStudio 的业务后端目录在哪里？
3. control 的前端和后端分别在哪里？
4. 新项目应该创建到哪个目录？
5. stories / 项目扫描优先扫描哪个目录？
6. 当前 Hermes 的权威输入文件有哪些？
7. `ai/memory/TASK_QUEUE.md` 和 `ai/memory/LONG_TASKS.md` 分别负责什么？
8. `apps/editor`、`apps/server`、`apps/control-server` 三者分别负责什么？
9. 如果扫描不到项目，最终回答必须至少包含哪三类信息？
10. control 和 Hermes 当前主线要完成的端到端任务链是什么？

### 第二段：小任务执行

目的：验证 Hermes 是否能把“知道事实”转成“正确行动”。

任务特点：

- 只允许 1 到 3 步
- 每步都有明确验收
- 不要求跨模块大范围推理

通过标准：

- 能先定位正确目录，再执行动作
- 不遗漏验证步骤
- 失败时能返回明确证据，而不是编造结果

推荐任务：

1. 找出图片生成相关的主要服务端入口文件。
2. 找出 control 中负责 Hermes 对话和 reasoning 的主要后端文件。
3. 找出 editor 中承载主应用入口的文件。
4. 扫描 `storage/projects`，说明当前有哪些项目。
5. 如果没有找到目标项目，返回已扫描目录、缺失目录和证据不足原因。

### 第三段：长任务 plan

目的：验证 Hermes 是否能在正确项目理解上做多步骤闭环。

前置条件：

- 第一段通过
- 第二段通过
- 已经有最小输入 contract

通过标准：

- 计划步骤不漂移到错误目录
- 计划中每一步都对应真实模块或真实文件
- 执行后能给出验证证据
- 失败时能停在明确边界，而不是继续幻想式推进

推荐长任务：

1. 创建故事项目骨架
2. 配置项目生成参数
3. 生成一份最小 story / scripts 结构
4. 说明哪些部分已完成，哪些部分仍缺 contract 或服务支持

## 测试执行规则

- 每轮只测一个维度，不要把问答、定位、执行、长 plan 混在一轮里。
- 每轮测试后都要写结论：
  - 哪些答案稳定正确
  - 哪些问题仍会漂移
  - 应该补哪份文档或记忆
- Hermes 回答错误时，优先修正 `README.md`、`ai/AGENTS.md`、`ai/MEMORY.md`、`ai/memory/STATUS.md`、`ai/memory/TASK_QUEUE.md`，而不是继续追问同一个问题很多轮。

## 当前建议顺序

1. 先做 10 题问答校准
2. 再做 5 个 1 到 3 步的小任务
3. 最后再做“创建项目 -> 配置 -> 生成文本”的长任务 plan

## 当前判断标准

- 如果第一段都不稳定，不要上长 plan。
- 如果第一段稳定、第二段不稳定，说明它“知道项目，但不会正确落动作”。
- 如果前两段稳定、第三段失真，说明问题不在项目理解，而在 plan 粒度、工具约束或 contract 缺失。
