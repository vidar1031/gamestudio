# P0 复盘（2026Q1）

本文用于回答两个问题：
1) 当前 P0 是否“可用且闭环”；
2) 哪些能力已进入 P1 范围，需转入结构化推进。

## 1. 结论

- 当前项目已经达到 **P0 可用闭环**：
  - 三层工作流可跑通（脚本/蓝图/合成）。
  - AI 写故事、AI 提示词、AI 分镜图（ComfyUI/SDWebUI）可在本地完成生产。
  - 资源可落盘、可预览、可导出、可回放。
- 但当前仍属于 **“功能可用，工程治理不足”** 状态：
  - 参数、状态、失败续跑、可观测性、文档规范近期开启补齐，但还未形成统一标准。
  - 多 Provider（openai/doubao/ollama/sdwebui/comfyui）联动复杂，缺少系统化测试基线。

## 2. 已具备能力（按链路）

### 2.1 文本链路
- 写故事 Provider：`local/openai/doubao/ollama`
- 提示词 Provider：`openai/doubao/ollama`
- 模型选择支持记忆与测试连接（设置侧）
- 公式校验与随机公式按钮已落地

### 2.2 图像链路
- Provider：`doubao/sdwebui/comfyui`
- 支持单场景出图与批量分镜出图
- 批量分镜支持：
  - 全局锚点 + 场景增量
  - 失败续跑（提示词/出图）
  - 场景级计时、运行日志
  - 持久化（重开弹窗不丢）
- ComfyUI：
  - 读取 checkpoint 模型列表
  - LoRA 预留配置（暂不生效）
  - 模型名自动规范化（兼容 `xxx [hash]`）

### 2.3 编辑器与资源
- 节点、角色、资源三栏工作流可用
- 资源属性支持：
  - 图片预览
  - 打开所在文件夹

## 3. 主要风险与缺口（P1 必做）

1. **测试缺口**：缺少端到端回归测试（尤其 AI/出图多 Provider 场景）。
2. **状态治理**：`project.state.aiBackground` 已较复杂，需 schema 化与迁移策略。
3. **可观测性**：日志已可用，但缺少统一 traceId、阶段指标、错误分层。
4. **配置治理**：设置项增多，缺少“分层配置优先级”文档与校验矩阵。
5. **开发约束**：注释风格、P1 边界、TODO 规范尚未统一。

## 4. P0 收口建议

- 以“冻结 P0 行为 + 建立 P1 约束”为目标，不再新增跨层能力。
- 新需求默认进入 P1 文档评审后再实现，避免继续在 P0 叠补丁。
