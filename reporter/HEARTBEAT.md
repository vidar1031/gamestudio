# HEARTBEAT - 汇报者的定期检查

## 轮询任务

当被触发轮询时，执行以下任务（仅在自动轮询中）：

### 1. 服务健康诊断
- 运行 `bash scripts/lifecycle/status_project.sh`
- 检查 Server、Editor、OpenClaw 运行状态
- 记录故障项（如有）

### 2. 任务进展汇总
- 读取 `memory/STATUS.md` 当前目标
- 统计 `memory/TASK_QUEUE.md` 中的待办/进行中/完成项
- 识别新的阻塞项

### 3. 今日进展记录
- 检查是否有新的日期日志（`memory/YYYY-MM-DD.md`）
- 记录今日完成项摘要

### 4. **代理功能检查**（可选轮询任务）
当需要验证代理是否按照自己的 md 真实工作时：

- 选择一个待检查的代理（或轮流检查）
- 触发该代理的会话，发送验证问题：
  ```
  "请用一句话介绍自己：你是谁？你的角色和工作方式是什么？"
  ```
- 记录回答内容
- 对比其 IDENTITY.md 和 SOUL.md：
  - 身份声明是否一致？
  - 角色描述是否相符？
  - 工作方式描述是否体现？
- 如果有偏差，标记为 ⚠️ 或 ❌ 并记录具体不一致之处
- 检查代理是否理解其他代理的职责：
  ```
  "你知道 Executor 是谁吗？他的工作是什么？"
  ```

## 输出格式

简要汇报（控制在 500 字以内）：

```
📊 GameStudio 项目状态
✅ 服务状态：Server 运行中、Editor 可访问
📋 进度：3 个待办 / 1 个进行中
🎯 当前目标：正式场景出图验证
🚧 阻塞项：资产漂移案例收集中
🧪 代理检查：Critic 身份验证 ✅ 一致
💡 建议：继续推进当前验证任务
```

当发现代理偏差时：

```
📊 GameStudio 项目状态
✅ 服务状态：正常
📋 进度：无变化
🔴 代理异常：Executor 身份声明与回答不符
   - 配置说：务实精准、不废话
   - 回答中：有冗长的思考过程描述
   - 建议：更新 executor/SOUL.md 中的工作哲学
🔧 下一步：需要与 executor 沟通并更新配置
```

## 重要说明

- 仅在自动轮询模式下执行
- 人类用户消息不触发轮询
- 每次轮询超时 10 分钟
- 代理功能检查是可选轮询任务（不是每次都必须做，可按需激活）
- 如发现严重问题（服务崩溃等）立即标记为高优先级
- 发现代理配置与实际行为不符时，立即标记为 RED FLAG 并建议改进
